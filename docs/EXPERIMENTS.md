## 2026-07-16 — 4주차 확장: alpha 그리드 스윕 & 청킹 전략 비교

### 배경
4주차 강의 스크립트를 코퍼스에 추가하고(88파일 / 5,120청크), 비대칭 임베딩(e5 query/passage
프리픽스)·코사인 거리공간·주차 메타데이터 필터·유니코드 전처리를 적용했다. 임베딩 방식이 바뀌었으므로
하이브리드 가중치 alpha를 재확정하고, 이어서 청킹 전략을 비교했다.

### 공통 환경
- 코퍼스: 1~4주차 강의 스크립트 88개 파일
- 임베딩: `intfloat/multilingual-e5-small` (query:/passage: 프리픽스), 코사인 거리(`hnsw:space=cosine`)
- 검색: BM25 + Vector 하이브리드 — `score = alpha·norm(vector) + (1-alpha)·norm(bm25)` (각 min-max 정규화)
- 골든셋: 26문항 (1~3주차 20 + 4주차 초안 6)
- 지표: Hit Rate@5, MRR, combined = 두 지표의 조화평균
- 재현 스크립트: `scripts/run_evaluation.py`(실험 1), `scripts/compare_chunking.py`(실험 2)

### 실험 1 — alpha 그리드 스윕 (0.0 ~ 1.0, 0.1 간격)
청킹 = block(기본). alpha는 벡터 검색 비중(1.0=벡터만, 0.0=BM25만).

| alpha | Hit Rate@5 | MRR | combined |
|---|---|---|---|
| 0.0 | 0.731 | 0.647 | 0.687 |
| 0.1 | 0.769 | 0.683 | 0.723 |
| 0.2 | 0.846 | 0.703 | 0.768 |
| 0.3 | 0.846 | 0.734 | 0.786 |
| 0.4 | 0.885 | 0.724 | 0.797 |
| **0.5** | **0.885** | **0.782** | **0.830** ← 선택 |
| 0.6 | 0.923 | 0.742 | 0.822 |
| 0.7 | 0.885 | 0.734 | 0.802 |
| 0.8 | 0.885 | 0.750 | 0.812 |
| 0.9 | 0.885 | 0.722 | 0.795 |
| 1.0 | 0.808 | 0.721 | 0.762 |

- **선택: alpha=0.5** (combined 최고). Hit@5=0.885, MRR=0.782 → AC-007(≥0.8, ≥0.5) PASS.
- 양 끝(0.0=BM25만, 1.0=벡터만)이 가장 낮음 → 하이브리드가 단일 방식보다 유효함을 재확인.
- 참고: 1~3주차만(20문항)일 때는 alpha=0.6·Hit@5=1.000이었으나, 4주차 문항 추가 후 0.885로 하락.

### 실험 2 — 청킹 전략 비교 (block / merged / tokens)
alpha=0.5 고정. 전략마다 chunk_id가 달라지므로 "검색된 청크 텍스트가 정답 블록 텍스트를 포함하는가"로 판정.

| 전략 | 청크 수 | Hit Rate@5 | MRR | 색인 시간 |
|---|---|---|---|---|
| block (타임스탬프 블록 1개=청크) | 5,120 | 0.885 | 0.782 | ~330s |
| merged (모드 A: 블록 3개 병합) | 1,735 | 0.923 | 0.761 | ~165s |
| tokens (모드 B: 토큰 256 예산) | 1,158 | 0.923 | 0.782 | ~118s |

- 블록을 묶어 문맥을 넓히면 **Hit Rate@5 0.885 → 0.923 개선**, 청크 수·색인 시간도 대폭 감소.
- **최적: tokens** — Hit·MRR 모두 최상이면서 청크 수 최소(비용 최소).
- 원인: 기본 block은 한 개념이 짧은 블록 여러 개에 흩어져 정답 블록을 top-5에 올리기 어려움
  (예: "HNSW … Navigable Small World"가 블록 경계에서 분리됨).

## 2026-07-16 — 관련성 게이트(할루시네이션 억제) 3종 시도와 한계

### 배경
로컬 LLM(Ollama)로 검색+생성(`scripts/ask.py`)을 붙이자, **코퍼스에 없는 질문**("오늘 날씨 어때?")에도
LLM이 억지로 답을 지어내는 할루시네이션이 나타났다. "자료로 답할 수 있는 질문만 통과시키는 관련성 게이트"를
세 방식으로 시도했고, 셋 다 이 코퍼스에선 실패했다. 그 원인을 기록한다.

### 공통 환경
- 코퍼스: 1~4주차 강의 스크립트(구어체 전사, 88파일 / 5,120 block 청크)
- 검색: 하이브리드 alpha=0.5, top_k=4
- 생성/판정 LLM: Ollama `gemma2:2b`, `gemma4:e2b` (로컬, GPU 없음)

### 시도 1 — 검색 점수 임계값 (벡터 거리 / BM25)
"관련 있으면 거리가 가깝다"를 가정하고 임계값으로 컷.

| 질문 | 코사인 거리(최상) | BM25(최상) |
|---|---|---|
| (관련) 강의 내용 질문 | ~0.11 | — |
| (무관) "오늘 서울 날씨 어때?" | **0.118** | **16.9 (최고)** |

- **실패**: 무관 질문의 거리(0.118)가 관련 질문(~0.11)과 사실상 겹침 → 분리 안 됨.
  임베딩 이방성(anisotropy)으로 무관한 문장도 거리가 비슷하게 나온다.
- BM25는 더 나빴다 — "오늘/서울/날씨"가 강의 예시 문장에 흔히 등장해 무관 질문이 오히려 최고점.

### 시도 2 — LLM-as-judge (예/아니오 판정)
답 생성 전, "이 자료로 답할 수 있나?"를 LLM에게 예/아니오로 물어 컷.

| 질문 | 기대 | gemma2:2b | gemma4:e2b |
|---|---|---|---|
| "오늘 날씨 어때?" (무관) | 차단 | **통과**(오답) | — |
| "벡터 인덱싱 알고리즘 두 가지는?" (관련) | 통과 | — | **차단**(오답) |
| "김치찌개 레시피?" (무관) | 차단 | 차단(정답) | 차단(정답) |

- **실패**: 작은 로컬 모델의 판정이 **양방향으로 다 틀린다**(무관을 통과, 관련을 차단). 판단 분산이 커 신뢰 불가.
- 함정 사례: "오늘 날씨"는 강의가 도구 사용 예시로 *"오늘 서울 날씨 어때?"*를 실제로 언급 → "언급 ≠ 답변"을
  작은 모델이 구분 못 함.

### 시도 3 — 리랭커(cross-encoder) 절대 점수 임계값
`BAAI/bge-reranker-base`로 (질문, 청크) 관련도 로짓을 매겨 임계값 컷. (원시 로짓 사용 — 기본 Sigmoid는
점수를 0~1로 뭉개 스케일이 사라짐, `activation_fn=Identity`로 해제)

| 질문 | 최고 로짓 | 판정 |
|---|---|---|
| (관련) 코사인 유사도가 뭐야? | **-5.79** | 관련 중 최고 |
| (관련) 청킹 전략에는? | -6.62 | |
| (관련) 벡터 인덱싱 알고리즘 두 가지는? | -7.26 | 관련 중 최저 |
| (무관) 손흥민 골 몇 개야? | **-4.00** | ← 무관인데 관련 전부보다 높음 |
| (무관) 오늘 날씨 어때? | -7.17 | 관련 범위에 섞임 |
| (무관) 김치찌개 레시피? | -10.19 | |

- **실패**: 무관 "손흥민"(-4.00)이 관련 질문 전부(-5.79~-7.26)보다 높아 **어떤 임계값도 분리 불가**.
- **근본 원인 = 청크 품질**(리랭커 자체가 아님):
  - 정답 강의(23차시 "HNSW vs IVF")의 해당 청크가 **"안녕하세요, 이번 시간에는…"** 인사말로 시작 → 리랭커가
    "답을 안 주는 조각"으로 보고 -10.16 부여.
  - "손흥민 골 몇 개" ↔ CoT 강의의 *"5개에서 10개 정도…"* 숫자 표현이 겹쳐 -4.00 오탐.
- **교훈**: *리랭커 성능은 청크 품질에 종속된다.* 정답이 인사말·잡담과 뒤섞인 시간순 전사 청크로는 좋은 리랭커도 못 살린다.

## 2026-07-22 — 5주차 확장: RAGAS Answer Relevancy 개선 시도

### 배경
5주차 확장(RAG 답변 생성 + RAGAS 평가) 도입 직후 golden_set 26문항 전부(ground_truth 채움)로
RAGAS를 처음 돌렸을 때 4대 지표 중 **Answer Relevancy(0.784)가 가장 낮았다**. 6-1 강의에서
배운 계산 원리(LLM이 답변을 보고 역질문을 생성해 원래 질문과의 코사인 유사도를 잰다)에 기반해,
`core/rag_chain.py`의 시스템 프롬프트만 바꿔가며 개선을 시도했다.

### 공통 환경
- 코퍼스·검색·재랭킹 파이프라인은 전혀 건드리지 않았다 (프롬프트 문자열만 변경)
- 골든셋: 26문항 전부 (ground_truth 있는 항목 전수 평가)
- 생성/판정 LLM: Anthropic Claude Haiku(`claude-haiku-4-5-20251001`), temperature=0.0
- 재현 스크립트: `scripts/run_ragas_evaluation.py`
- 지표: RAGAS Faithfulness / Answer Relevancy / Context Precision / Context Recall

### 결과

| 지표 | 원본 | 시도1: "간결하게 답하라" | 시도2: "질문 주제를 되풀이하라" |
|---|---|---|---|
| Faithfulness | 0.877 | **0.902** | 0.878 |
| Answer Relevancy | 0.784 | 0.748 (↓) | **0.790 (↑)** |
| Context Precision | 0.844 | 0.844 | 0.844 |
| Context Recall | 0.923 | 0.923 | 0.962 |

**시도1 — "질문에서 실제로 묻는 것에 직접적이고 간결하게 답하라"**: 의도와 반대로 Answer
Relevancy가 **0.784 → 0.748로 오히려 하락**했다. 가설: 답변이 짧아지면서 질문의 핵심 주제/표현이
답변 문장에서 빠져, RAGAS가 역질문을 생성할 때 참고할 단서가 줄어 원래 질문과의 유사도가 낮아진
것으로 보인다. 다만 Faithfulness는 0.877→0.902로 개선됐다 — 부연 설명이 줄면서 근거 없는 내용을
덧붙일 여지도 함께 줄어든 것으로 추정된다.

**시도2 — "질문에서 묻는 대상(핵심 주제·표현)을 답변 문장에 되풀이하며 직접 답하라"**(예시 포함,
"국내 출장 숙박비 한도는?" → "국내 출장 숙박비 한도는 15만 원입니다." 형식 제시): Answer
Relevancy가 0.784→0.790으로 소폭 개선됐고 Faithfulness도 원래 수준(0.878)을 유지했다.

### 한계 — 노이즈와 신호를 구분하기 어렵다
Context Recall이 검색 파이프라인을 전혀 안 건드렸는데도 0.923(시도1)→0.962(시도2)로 흔들렸다.
이는 RAGAS의 LLM-as-judge 계산 자체에 run-to-run 변동성이 있다는 뜻이다. 시도2의 Answer
Relevancy 개선폭(+0.006)은 이 노이즈 폭(Context Recall에서 관측된 ±0.04 수준)보다 훨씬 작아서,
**"확실히 개선됐다"고 단정하기보다 "퇴보하지 않은 최선의 시도"** 정도로 해석하는 것이 정직하다.
26문항이라는 표본 크기도 이 판단을 정밀하게 하기엔 넉넉하지 않다.

## 2026-07-22 — 4주차 골든 6문항(gs-21~26) 검토

## 2026-07-22 — gs-23 회귀 원인 심층 분석

### 배경
청킹 전략 채택(위 항목) 이후 gs-23이 top-5에서 새로 누락됐다. `core.chunking.chunk_script()`로
정답 청크(10차시, 이론 강의)와 1·2위 경쟁 청크(13차시, 실습)의 tokens 전략 원문을 block
전략 원문과 나란히 대조했다.

### 발견
10차시(이론)와 13차시(실습)는 **같은 개념("asymmetric embedding")을 서로 다른 밀도로
설명**하고 있었다:

- **정답 청크(10차시, 이론)**: "비대칭 임베딩, asymmetric 임베딩이라는 개념인데요. 보통
  쿼리는 짧은 자연어 질문... 문서 청크는 수백 토큰..." — 개념을 일반적 수준에서 설명.
- **경쟁 청크(13차시, 실습)**: "이게 asymmetric embedding... 문서를 저장할 때와 쿼리로
  검색할 때 서로 다른 임베딩 방식... 실제로 코드를 보시면 `get_embedding` 함수에
  `is_query`라는 인자가 있죠. `is_query`는 `False`가 기본값인데..." — 같은 개념을
  **코드 수준까지 구체적으로** 설명.

block 전략에서는 13차시의 이 코드 설명이 문장 하나씩(13차시_1, _8, _10 등) 잘게 쪼개져
있어 각 조각의 관련성 신호가 희석되어 있었다. tokens 전략(256토큰 병합)으로 바뀌면서
"asymmetric embedding 언급 + 코드 설명"이 한 청크(13차시_1, _2) 안에 조밀하게 뭉쳐졌고,
그 결과 검색 신호가 이론 강의 청크보다 강해져 순위를 역전시켰다.

### 결론 — 이것은 검색 실패가 아니라 골든셋 평가 방식의 한계다
경쟁 청크(13차시)는 오답이 아니라 **같은 질문에 대한 더 구체적인 정답**이다. 사용자가 이
청크를 받아도 "쿼리와 문서를 서로 다른 방식으로 임베딩하는 개념의 이름"에 정확히 답할 수
있다. 즉 이 코퍼스는 같은 개념을 이론 강의와 실습 세션 두 곳에서 설명하는데, golden_set의
`expected_chunk` 필드는 "정답은 정확히 이 청크 하나뿐"이라고 가정하는 단일 정답 스키마다.
Hit Rate@5를 `chunk_id` 정확히 일치로 판정하는 현재 방식(`core/evaluation.py`)은 이런
"복수의 유효한 정답 소스" 상황을 구조적으로 반영하지 못한다 — `scripts/compare_chunking.py`가
이미 이 문제를 인지하고 substring 포함 여부로 판정 방식을 바꿨던 것과 같은 종류의 한계다.

검토했지만 **추가하지 않은** 애매한 경우도 있다: gs-18("함수를 그럴싸하게 호출하는 환각")과
비슷하게 "hallucination"을 언급하는 청크(Tool Selection 강의)가 있었지만, 그건 "존재하지
않는 tool을 잘못 선택하는" 다른 맥락이라 "함수 호출" 질문과 정확히 같은 주장은 아니라고
판단해 제외했다. gs-20의 rank-1 경쟁 청크도 같은 학습 원리(positive는 가깝게, negative는
멀게)를 설명하지만 "Triplet Loss"라는 이름 자체는 언급하지 않아 제외하고, 이름을 명시한

## 2026-07-22 — golden_set 15문항 확장 (26→41)

### 배경
데이터 품질(전처리)에 더 투자하면 성능이 더 좋아질지에 대한 논의에서, 현재 Hit Rate@5=0.962/
MRR=0.872는 26문항짜리 골든셋 기준 이미 천장에 가깝다는 점, 그리고 코퍼스 88개 파일 중
정작 골든셋이 정답 청크로 참조하는 파일은 24개뿐 — 나머지 **64개 파일은 단 1문항도 없어서
검증이 안 되고 있다**는 점을 확인했다. 전처리보다 먼저 골든셋 자체의 커버리지를 넓히는 쪽을
선택했다.

### 방법
미커버 64개 파일 중 4개 주차에 고르게 분산한 15개 파일(1주차 4, 2주차 4, 3주차 4, 4주차 3)을
선정. 각 파일을 실제 `core.chunking.chunk_script()`로 청킹한 원문을 직접 읽고, 하나의 정확한
사실을 묻는 질문을 작성한 뒤 `core.scoring.execute_hybrid_search`(top_n=10)로 실제 정답
순위를 검증했다.

검증 과정에서 애초에 의도한 청크가 아니라 **인접 청크에 답이 나뉘어 있거나 다른 차시에 더
정확한 설명이 있는 경우**를 3건 발견해 반영했다:
- **gs-28** (Prompt Analyzer Hook의 역할 분리 원칙): 의도한 청크(`_5`)가 6위였는데, 바로
  앞뒤 청크(`_4`, `_6`)가 같은 설명을 나눠 담고 있어 셋 다 정답으로 등록(4/5/6위 분포).
- **gs-29** (Stop Hook을 Command→Prompt로 전환한 이유): 의도했던 요약 청크(`18차시_...
  quality check_19`)가 top-10에 아예 안 잡혔다. 대신 같은 이유를 설명하는 청크가 **1주차
  다른 강의**("13차시_ 3-3. 용어 설명 part 1_4" — Command 타입의 한계)와 원래 강의
  ("18차시_...quality check_6" — 정규식 오탐/미탐)에 나뉘어 있는 것을 확인, 둘 다 정답으로
  등록. gs-23(비대칭 임베딩, 이론+실습)과 같은 패턴이다.
- **gs-33** (KV Cache로 시간복잡도가 O(n²)→?): 같은 사실을 담은 청크가 두 개(`_5`, `_7`)
  있어 둘 다 등록.
- **gs-35** (BLEU/ROUGE가 각각 무엇을 중심으로 측정하는가): 강의 중 같은 문장이 이론 설명
  때(`_6`)와 실습 시작 전 복습(`_11`)에서 두 번 나와 둘 다 등록.

**의도적으로 손대지 않은 것: gs-34** (Structured Output의 validation 두 가지 접근법)는
1단계 하이브리드 검색만 놓고 보면 정답 청크가 6위로 top-5를 놓친다. 질문을 다른 청크가
걸리도록 바꾸거나 청크를 추가로 끼워 맞추지 않고 **있는 그대로 두었다** — 골든셋은
파이프라인이 잘하는 걸 보여주기 위한 것이 아니라 약점을 드러내기 위한 것이므로. (참고로
실제 프로덕션 파이프라인은 여기서 끝나지 않고 Cross-Encoder 재랭킹을 한 번 더 거치는데,
`retrieve_and_rerank`로 확인해 보면 이 항목도 재랭킹 후에는 1위로 복구된다 — 아래 결론 참고.)

### 결과
alpha 재스윕 결과 최적값이 0.4 → **0.5**로 이동했다(41문항 기준 재확정):

| 지표 | 26문항(alpha=0.4) | 41문항(alpha=0.5) |
|---|---|---|
| Hit Rate@5 | 0.962 | 0.976 |
| MRR | 0.872 | 0.859 |

alpha가 0.5로 이동하면서 새로 생긴 트레이드오프도 있다: 기존 항목 **gs-26**("Chroma가
메타데이터 저장에 쓰는 내장 DB는?")이 **1단계 하이브리드 검색만 놓고 보면** alpha=0.4에서는
hit이었는데 alpha=0.5에서는 top-5를 놓친다(벡터 유사도 비중이 커지면서 다른 ChromaDB 실습
청크들의 순위가 더 높게 나옴). 하지만 실제 프로덕션 파이프라인(`retrieve_and_rerank`,
hybrid_top_n=20 → Cross-Encoder 재랭킹 → top-5)으로 확인해 보면 정답 청크가 다시 **1위로
복구**된다 — Cross-Encoder가 1단계의 순위 흔들림을 보정해 준 것이다. 즉 alpha=0.5로 인한
gs-26의 "회귀"는 실사용자에게는 영향을 주지 않는, 재랭킹 이전 단계에 한정된 현상이다.

MRR이 0.872→0.859로 소폭 하락한 이유: 새로 추가된 15문항 중 다수(gs-31/32/33/36/37/38/39/40/41)가
1위로 hit하지만, gs-28/29/34처럼 순위가 낮은 항목들이 평균을 끌어내렸다. 표본이 26개에서
41개로 늘면서 파이프라인의 실제 성능 분포가 더 정직하게 드러난 것으로 해석한다 — 문항 수가
적을 때는 우연히 잘 맞는 소수 항목에 지표가 좌우되기 쉽다.
